**关键教训：hash 同步陷阱（最重要）**

服务器在**处理**完一个 batch 时才消耗 hash（不是返回时）。正确的提交流程：

```
提交 batch → 服务器处理 → 返回 {hash: NEW_HASH, nextBatch: [...]} → 
重新调用 GET /api/exam/status?id=<examId> 获取最新 hash → 
用最新 hash 提交下一 batch
```

绝对不要信任 batch 响应里直接返回的 hash——它可能与服务器端的实际状态不同步，导致 `Invalid hash` 错误。必须从 status API 重新获取。
